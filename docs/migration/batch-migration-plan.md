# Flutter 迁移分批计划

本文用于后续逐批执行 Flutter 迁移。每一批都应保持可编译、可测试、可回滚。

## Batch 0：技术底座收口

- 目标：把当前 Flutter 工程从“可运行骨架”收口到可靠底座。
- 内容：
  - 修复 `review-fix-todo.md` 中 P1/P2 问题。
  - 明确并冻结首版表结构。
  - 补齐备份 v1 payload。
  - 为核心 Repository 增加失败场景测试。
- 验收：
  - `flutter analyze` 通过。
  - `flutter test` 通过。
  - `flutter build ios --simulator` 通过。

## Batch 1：设计系统与 App 壳

- 目标：把 HTML 原型和 `DESIGN.md` 落到 Flutter 可复用组件。
- 内容：
  - 主题 token、字体层级、颜色、圆角、阴影、间距统一收口。
  - 实现基础组件：页面容器、业务卡片、状态标签、底部导航、底部弹层、表单项、分段控件、主按钮、危险按钮。
  - 三个主 Tab 保持稳定：提醒 / 记录 / 我的。
  - 新增记录作为全局高频入口预留。
- 验收：
  - 组件示例页或测试页面可覆盖主要组件状态。
  - iOS simulator 下视觉与原型主基调一致。

## Batch 2：车辆与当前应用车辆

- 目标：先建立所有业务页面共享的车辆上下文。
- 内容：
  - 车辆列表、新增车辆、编辑车辆、删除车辆。
  - 当前应用车辆选择与失效回退。
  - 新车初始化保养项目配置。
  - 同一品牌 + 车型唯一规则。
- 验收：
  - 删除当前应用车辆后能稳定回退。
  - 删除车辆时关联项目和记录清理策略通过测试。

## Batch 3：保养项目配置

- 目标：完成默认项目、自定义项目、启用/禁用和提醒参数配置。
- 内容：
  - 默认车型项目 catalog。
  - 默认项目复制到车辆维度。
  - 自定义项目新增、编辑、删除限制。
  - 项目名称同车不重复。
  - 至少保留一个可用项目。
- 验收：
  - 默认项目使用 `catalogKey` 作为稳定身份。
  - 有历史记录的自定义项目不能删除。

## Batch 4：保养记录

- 目标：完成记录新增、编辑、删除和列表查询。
- 内容：
  - 新增记录：日期、里程、费用、项目、备注。
  - 同车同日唯一。
  - 同车同日同项目唯一。
  - 记录保存后车辆当前里程只增不减。
  - 按周期展示和按项目展示的数据查询。
- 验收：
  - 重复日期、重复项目、跨车项目引用均被拦截。
  - 删除单条记录或单个项目后，列表和项目维度视图一致。

## Batch 5：保养提醒

- 目标：完成提醒页业务计算和展示。
- 内容：
  - 当前应用车辆范围内计算提醒。
  - 最近保养记录索引。
  - 里程进度和时间进度谁先到采用谁。
  - 黄色 `>=100% && <125%`，红色 `>=125%`。
  - 无历史记录和无可用项目空状态。
- 验收：
  - 仅里程、仅时间、里程+时间、无历史记录场景都有测试。
  - 提醒排序和状态展示稳定。

## Batch 6：个人中心与数据能力

- 目标：完成本地工具型 App 的管理能力。
- 内容：
  - 当前应用车辆入口。
  - JSON 备份导出。
  - JSON 恢复导入。
  - 清空数据。
  - 手动日期开关和日期选择。
- 验收：
  - 导入失败必须回滚。
  - 手动日期影响提醒计算和车龄口径。

## Batch 7：iOS 首版打磨

- 目标：以 iOS 为优先平台完成首版可用性。
- 内容：
  - iOS 权限、文件导入导出、分享面板。
  - 启动性能和冷启动检查。
  - 暗色模式是否支持另行确认；默认先按浅色实现。
  - App 图标、启动页、Bundle ID、签名配置。
- 验收：
  - 真机或模拟器完整走通核心链路。
  - `flutter build ios` 或发布目标构建通过。

## Batch 8：Android 补齐

- 目标：在 iOS 稳定后补 Android 工具链和平台适配。
- 内容：
  - 安装 Android SDK / Android Studio。
  - 文件导入导出适配 Android。
  - 返回键、系统导航栏、安全区、键盘遮挡检查。
  - Android 构建和基础真机测试。
- 验收：
  - `flutter doctor` Android toolchain 正常。
  - `flutter build apk` 或目标格式构建通过。
