# car-record 业务逻辑 PRD

## 1. 文档定位

本文档用于沉淀 `car-record` 当前版本的业务逻辑，作为后续重构、重新设计 UI/交互、甚至更换开发语言时的产品规则依据。

本文档只描述业务对象、业务流程、数据口径和必须保留的约束；不要求未来实现继续沿用当前 SwiftUI、SwiftData、目录结构或页面组件。

### 1.1 当前代码事实

- 当前产品是纯本地 iOS 客户端。
- 当前没有账号、登录、云同步、网络接口或服务端依赖。
- 当前业务数据默认保存在用户设备本地。
- 当前主入口包含三个业务区：保养提醒、保养记录、个人中心。
- 当前支持应用内 JSON 备份与恢复。
- 当前支持调试用“手动日期”，影响今天、车龄、提醒进度等日期相关逻辑。

### 1.2 重构时必须保留

- 业务日期统一按日期维度处理，格式为 `yyyy-MM-dd`。
- 如果底层数据模型必须带时间，业务日期对应时间固定为当天 `00:00:00`。
- 不得把展示文案作为稳定主键；保养项目必须优先使用稳定 ID 或 `catalogKey` 关联。
- 车辆、保养项目、保养记录之间的隔离和唯一性规则必须保留。

## 2. 产品概览与业务边界

`car-record` 是面向个人车主的本地车辆保养记录工具。用户可以维护自己的车辆、配置保养项目、记录每次保养，并根据车辆当前里程和上次保养情况查看后续保养提醒。

### 2.1 核心能力

- 管理车辆基础信息。
- 为每辆车维护保养项目配置。
- 记录车辆每次保养的日期、里程、费用、项目和备注。
- 按当前应用车辆查看保养提醒。
- 按当前应用车辆查看和管理保养记录。
- 导出本地业务数据为 JSON 备份文件。
- 从 JSON 备份文件恢复业务数据。

### 2.2 不在当前业务范围内

- 账号体系、登录注册、会员体系。
- 云端同步、多设备同步、服务端备份。
- 支付、订单、门店、维修厂预约。
- OCR、发票识别、图片附件、保养单扫描。
- 多用户协作或家庭共享。

## 3. 核心业务对象

### 3.1 车辆

车辆表示一辆用户维护的车。

业务字段：

- 车辆 ID：稳定唯一标识。
- 品牌：例如“本田”“日产”。
- 车型：例如“22款思域”“22款轩逸”。
- 当前里程：单位为公里，按整数保存。
- 上路日期：只按日期维度保存。
- 禁用保养项目列表：保存该车被用户关闭的保养项目 ID。

当前代码事实：

- 同一品牌加车型目前只允许存在一辆车。
- 当前品牌选项为“本田”“日产”。
- 当前本田车型选项为“22款思域”“23款CR-V”“24款雅阁”。
- 当前日产车型选项为“22款轩逸”“23款奇骏”“24款天籁”。
- 当前只有“本田 22款思域”和“日产 22款轩逸”有专属默认保养项目配置；其他车型使用兜底配置。

重构时必须保留：

- 车辆 ID 必须是稳定标识。
- 车辆当前里程不能因为新增保养记录而回退。
- 删除车辆必须同时清理该车关联的保养记录和保养项目配置。

### 3.2 保养项目

保养项目表示一类可被记录和提醒的保养内容，例如机油、空调滤芯、刹车油。

业务字段：

- 项目 ID：稳定唯一标识。
- 项目名称：用于展示，可被修改。
- 归属车辆 ID：项目只属于某一辆车。
- 是否默认项目：区分车型内置项目和用户自定义项目。
- 默认项目键 `catalogKey`：默认项目的稳定业务键。
- 是否按里程提醒。
- 里程提醒间隔：单位为公里。
- 是否按时间提醒。
- 时间提醒间隔：单位为月。
- 进度黄色阈值。
- 进度红色阈值。
- 创建时间。

当前代码事实：

- 所有保养项目都按车辆隔离。
- 默认项目和自定义项目都持久化为同一种项目配置。
- 默认项目使用 `catalogKey` 表示稳定身份。
- 自定义项目没有 `catalogKey`。
- 当前默认黄色阈值为 `100%`。
- 当前默认红色阈值为 `125%`。

重构时必须保留：

- 项目名称只能作为展示文案，不能作为稳定主键。
- 默认项目涉及规则映射、排序、导入导出时必须优先使用 `catalogKey`。
- 自定义项目如果已有历史保养记录，不能删除。
- 每辆车至少应保留一个可用保养项目。
- 单个项目必须至少开启一种提醒方式：按里程或按时间。

### 3.3 保养记录

保养记录表示某辆车在某一天发生的一次保养。

业务字段：

- 记录 ID：稳定唯一标识。
- 所属车辆。
- 保养日期：只按日期维度保存。
- 保养项目 ID 列表。
- 费用：非负数，最多保留两位小数。
- 保养发生时里程：单位为公里，按整数保存。
- 备注。

当前代码事实：

- 一条保养记录可以包含多个保养项目。
- 保养项目列表按项目 ID 保存，不按项目名称保存。
- “同车同日唯一”由 `车 ID + yyyy-MM-dd` 口径保证。
- “同车同日同项目唯一”由 `车 ID + yyyy-MM-dd + 项目 ID` 口径保证。

重构时必须保留：

- 同一车辆同一天只能有一条保养记录。
- 单条保养记录内的项目必须去重。
- 保养记录必须能按项目维度拆分查看。
- 删除车辆时，车辆关联的保养记录必须被清理。

### 3.4 当前应用车辆

当前应用车辆表示用户当前正在查看和维护的车辆上下文。

当前代码事实：

- 保养提醒页只展示当前应用车辆。
- 保养记录页只展示当前应用车辆。
- 新增保养记录默认限制在当前应用车辆范围内。
- 如果持久化的当前应用车辆 ID 失效，会回退到第一辆可用车辆。

重构时必须保留：

- 当前应用车辆是跨模块共享业务上下文。
- 提醒、记录、新增保养记录必须尊重当前应用车辆。
- 删除当前应用车辆后，必须有明确回退策略。

### 3.5 手动日期

手动日期是调试能力，用于临时替代系统今天。

当前代码事实：

- 手动日期开启后，“今天”相关业务逻辑使用手动日期。
- 手动日期会影响车龄、保养提醒时间进度等计算。
- 手动日期按当天 `00:00:00` 归一化。

重构时必须保留：

- 所有“今天”“当前日期”“车龄”“提醒进度”逻辑必须经过统一日期上下文。
- 业务逻辑中不要散落直接读取系统当前时间的实现。

## 4. 车辆与车型管理

### 4.1 新增车辆

用户新增车辆时，需要填写或选择：

- 品牌。
- 车型。
- 当前里程。
- 上路日期。
- 初始保养项目配置。

业务规则：

- 品牌不能为空。
- 车型不能为空。
- 当前里程按公里保存。
- 上路日期只保存日期。
- 同一品牌加车型目前不允许重复添加。
- 新车创建时，需要为该车生成一份独立的保养项目配置。
- 用户可以在新增车辆时启用、禁用、修改默认保养项目，也可以新增自定义项目。
- 保存前至少要有一个启用的保养项目。
- 保养项目名称在同一车辆配置内不能重复。

### 4.2 编辑车辆

用户编辑车辆时，可以修改：

- 当前里程。
- 上路日期。
- 保养项目配置。

当前代码事实：

- 编辑车辆时，品牌和车型以当前车辆为基础显示。
- 保养项目配置从该车辆已有配置读取。
- 如果切换车型，默认项目可见范围会按新车型配置重建。

业务规则：

- 编辑车辆时仍必须至少保留一个可用保养项目。
- 禁用项目不会出现在新增保养记录的可选项目中。
- 编辑已有保养项目时，不能出现空名称、重名项目或没有提醒方式的项目。
- 删除自定义项目前必须检查是否已有历史记录；有历史记录则禁止删除。

### 4.3 删除车辆

用户删除车辆时，需要二次确认。

业务影响：

- 删除该车辆。
- 删除该车辆关联的全部保养记录。
- 删除该车辆归属的全部保养项目配置。
- 如果删除的是当前应用车辆，需要重新解析当前应用车辆。

重构时必须保留：

- 删除是不可恢复操作，除非未来另行设计回收站或撤销机制。
- 删除车辆不能留下孤立的保养记录或保养项目配置。

## 5. 保养项目配置

### 5.1 默认项目

默认项目来自车型配置，用于给车辆创建初始保养项目。

当前内置配置：

- 本田 22款思域：
  - 汽油发动机清洁剂（燃油宝）：5000 公里。
  - 机油、机滤：5000 公里或 6 个月。
  - 空调滤芯：20000 公里或 12 个月。
  - 空气滤芯：20000 公里。
  - 变速箱油：40000 公里或 24 个月。
  - 制动液（刹车油）：36 个月。
  - 火花塞：100000 公里。
  - 检查传动皮带：40000 公里或 24 个月。
  - 检查气门间隙：120000 公里。
  - 检查制动器（刹车）：120000 公里。
  - 冷却液（防冻液）：200000 公里或 120 个月。
  - 汽油滤芯：140000 公里。
  - 轮胎换位：10000 公里。
- 日产 22款轩逸：
  - 机油：10000 公里或 6 个月。
  - 空调滤芯：20000 公里或 12 个月。
  - 空气滤芯：20000 公里。
  - 变速箱油：24 个月。
  - 刹车油：36 个月。

业务规则：

- 默认项目顺序用于自然排序。
- 默认项目可被用户禁用。
- 默认项目名称可以作为展示文案修改，但规则映射不能依赖名称。
- 默认项目恢复默认时，应按车型配置恢复提醒方式和间隔。

### 5.2 自定义项目

用户可以为车辆新增自定义保养项目。

业务规则：

- 自定义项目名称不能为空。
- 同一车辆内项目名称不能重复。
- 自定义项目必须至少开启一种提醒方式。
- 自定义项目默认按里程提醒，默认间隔 5000 公里。
- 自定义项目无历史保养记录时可以删除。
- 自定义项目已有历史保养记录时禁止删除。

### 5.3 启用与禁用

用户可以禁用某个保养项目。

业务含义：

- 禁用项目不会在新增保养记录时展示。
- 禁用项目不会在保养提醒中展示。
- 编辑历史记录时，当前代码会允许包含已禁用项目，以避免历史数据无法编辑。

重构时必须保留：

- 禁用是车辆维度配置。
- 禁用不能删除历史记录。
- 禁用不能破坏项目 ID 与历史记录之间的关联。

### 5.4 提醒规则配置

每个保养项目可以配置：

- 按里程提醒：使用保养间隔公里数计算进度。
- 按时间提醒：使用保养间隔月数计算进度。
- 同时按里程和时间提醒：谁更接近或超过到期，就采用谁作为主进度。

业务规则：

- 至少开启一种提醒方式。
- 里程间隔必须大于 0。
- 时间间隔必须大于 0。
- 黄色阈值当前为 `>=100% 且 <125%`。
- 红色阈值当前为 `>=125%`。

## 6. 保养提醒

保养提醒用于帮助用户判断当前应用车辆各保养项目距离下次保养的状态。

### 6.1 数据范围

业务规则：

- 只计算当前应用车辆。
- 只计算该车可见且未禁用的保养项目。
- 只使用该车的保养记录。
- 如果没有当前应用车辆，提醒为空。
- 当前代码事实：如果车辆没有任何保养记录，提醒页当前不生成车辆提醒分组。

### 6.2 最近保养记录

计算提醒时，需要找到每个保养项目最近一次保养记录。

业务规则：

- 按车辆和项目建立最近记录索引。
- 一条保养记录包含多个项目时，该记录同时作为这些项目的候选最近记录。
- 如果同一项目有多条记录，选择日期更晚的记录。
- 如果日期相同，选择里程更高的记录。
- 如果某项目没有历史保养记录，使用车辆上路日期作为时间基准，使用 `0` 作为里程基准。

### 6.3 里程进度

按里程提醒时：

- 已使用里程 = 当前车辆里程 - 最近一次该项目保养时里程。
- 已使用里程不能小于 0。
- 里程进度 = 已使用里程 / 项目里程间隔。
- 剩余里程 = 项目里程间隔 - 已使用里程。

展示口径：

- 剩余里程大于 0：距离下次约多少公里。
- 剩余里程等于 0：今日到期。
- 剩余里程小于 0：已超多少公里。

### 6.4 时间进度

按时间提醒时：

- 时间基准 = 最近一次该项目保养日期；如果没有记录，则为车辆上路日期。
- 到期日期 = 时间基准 + 项目时间间隔月数。
- 时间进度 = 从时间基准到今天的已过时间 / 从时间基准到到期日期的总时间。
- 剩余天数 = 今天到到期日期的自然日差。

展示口径：

- 剩余天数大于 0：距离下次约多少天、多少个月或多少年。
- 剩余天数等于 0：今日到期。
- 剩余天数小于 0：已超多少天、多少个月或多少年。

### 6.5 综合进度与排序

业务规则：

- 同时开启里程和时间提醒时，取进度更高的一项作为综合进度。
- 只开启里程时，使用里程进度。
- 只开启时间时，使用时间进度。
- 都未开启时，进度为 0，并显示未设置提醒规则。
- 进度展示百分比按综合进度四舍五入。
- 进度条展示值限制在 `0%` 到 `100%`，但文本百分比可以超过 `100%`。
- 提醒列表按综合进度从高到低排序。
- 综合进度相同时，按到期优先级排序。
- 仍相同时，按项目名称排序。

### 6.6 颜色状态

业务规则：

- 正常：进度 `<100%`。
- 黄色：进度 `>=100%` 且 `<125%`。
- 红色：进度 `>=125%`。

## 7. 保养记录

保养记录模块用于新增、查看、编辑和删除保养历史。

### 7.1 新增保养记录

用户新增保养记录时，需要填写：

- 车辆。
- 保养日期。
- 保养项目，可多选。
- 费用。
- 保养发生时里程。
- 备注。

业务规则：

- 默认车辆为当前应用车辆。
- 当前代码默认限制只能给当前应用车辆新增记录。
- 保养项目只能从该车辆可见且未禁用项目中选择。
- 至少选择一个保养项目。
- 费用必须是非负数，最多两位小数。
- 新增保存前需要确认本次保养项目的下次提醒间隔。
- 保存时同时保存保养记录，并可把确认页的间隔回写为项目默认提醒间隔。
- 如果同车同日已存在记录，禁止新增，并提示用户去记录页编辑该日期记录。
- 保存成功后，如果本次保养里程大于车辆当前里程，则更新车辆当前里程。
- 保存成功后，如果本次保养里程小于或等于车辆当前里程，车辆当前里程不回退。
- 新增记录并确认间隔后，当前代码会跳转到保养提醒。

### 7.2 编辑保养记录

编辑整条记录时，可以修改：

- 保养日期。
- 保养项目集合。
- 费用。
- 保养发生时里程。
- 备注。
- 所属车辆范围内的车辆。

业务规则：

- 编辑时如果目标同车同日已存在其他记录，禁止保存。
- 保存时同步维护“同车同日”和“同车同日同项目”唯一关系。
- 编辑时只有草稿有实际变化才允许提交。
- 保存成功后，车辆当前里程仍只允许被更大的记录里程抬高，不回退。

### 7.3 按项目入口编辑与拆单

用户从“按项目”视角编辑某个项目对应的记录时，存在特殊规则。

业务规则：

- 如果原记录只包含该一个项目，编辑该记录本身。
- 如果原记录包含多个项目，且编辑后仍在原同车同日周期内，则仍编辑原记录本身，但项目集合保持原集合。
- 如果原记录包含多个项目，且该项目被改到其他日期周期，则从原记录中移除该项目，并为该项目新建一条单项目记录。
- 拆单时，原记录保留剩余项目。
- 拆单新记录的费用和备注使用当前表单填写值。
- 拆单时仍必须遵守同车同日唯一规则。

### 7.4 按周期展示

按周期展示以保养日期为聚合单位。

业务规则：

- 数据范围为当前应用车辆。
- 按自然日聚合记录。
- 分组按日期倒序排列。
- 同日内记录按日期、里程、ID 排序。
- 分组展示项目摘要，项目按车型默认顺序排序。
- 分组总费用为该日记录费用之和。
- 支持按年份筛选。
- 支持按车辆筛选；当前应用车辆上下文下通常只有一辆。
- 支持按项目筛选。

### 7.5 按项目展示

按项目展示把一条多项目保养记录拆成多个项目行展示。

业务规则：

- 数据范围为当前应用车辆。
- 一条记录包含多个项目时，每个项目生成一行。
- 行按记录日期倒序、里程倒序、项目名称、行 ID 排序。
- 支持按项目筛选。
- 从项目行编辑时，进入按项目入口编辑规则。
- 删除项目行时，如果原记录只剩一个项目，则删除整条记录。
- 删除项目行时，如果原记录包含多个项目，则只从该记录移除当前项目，并同步唯一关系。

### 7.6 删除保养记录

业务规则：

- 删除整条记录会删除该记录及其项目关系。
- 从项目维度删除时，按 7.5 的拆项删除规则处理。
- 删除后如果当前正在编辑被删除记录，需要关闭编辑状态。
- 删除保养记录不会自动回退车辆当前里程。

## 8. 个人中心

个人中心承载车辆管理、当前应用车辆、备份恢复和调试能力。

### 8.1 当前应用车辆

用户可以在个人中心选择某辆车作为当前应用车辆。

业务规则：

- 被应用的车辆会成为保养提醒、保养记录和新增保养记录的默认业务范围。
- 当前应用车辆 ID 本地持久化。
- 当前应用车辆失效后，回退到第一辆可用车辆。
- 如果没有车辆，则当前应用车辆为空。

### 8.2 数据备份

用户可以导出当前全部业务数据。

导出范围：

- 全部车辆。
- 每辆车关联的全部保养记录。
- 车型保养项目配置快照。

业务规则：

- 导出为 JSON 文件。
- 导出文件名包含时间戳。
- 日期字段只导出 `yyyy-MM-dd`。
- 保养记录项目导出时，默认项目优先导出 `catalogKey`，没有 `catalogKey` 的自定义项目回退导出名称。
- 每个车型配置按品牌加车型去重导出。

### 8.3 数据恢复

用户可以从 JSON 备份恢复数据。

业务规则：

- 如果当前已有业务数据，恢复前需要二次确认。
- 恢复流程为先清空当前全部业务数据，再导入备份数据。
- 恢复失败时需要回滚数据库变更。
- 恢复成功后展示导入统计。
- 恢复失败时提示备份文件完整性或结构问题。

### 8.4 清空数据

用户可以重置全部业务数据。

清空范围：

- 全部车辆。
- 全部保养项目。
- 全部保养记录。

业务规则：

- 清空前需要二次确认。
- 清空后当前应用车辆置空。
- 清空是不可恢复操作，除非用户此前已有备份文件。

### 8.5 调试模式与手动日期

当前代码事实：

- 连续点击版本号可切换调试模式。
- 调试模式开启后，可以使用时间临时设置和控制台日志。
- 手动日期开启后，会影响全局业务“今天”。

重构时建议：

- 调试能力不作为普通用户核心业务。
- 如果新版本继续保留调试能力，必须避免影响正式用户数据语义。

## 9. 数据导入导出

### 9.1 JSON 顶层结构

当前备份 JSON 顶层包含：

- `modelProfiles`：车型保养项目配置数组。
- `vehicles`：车辆数组，每辆车包含车辆信息和保养记录。

### 9.2 车型配置结构

每个车型配置包含：

- 品牌。
- 车型。
- 保养项目数组。

每个保养项目包含：

- 项目 ID。
- 名称。
- 是否默认项目。
- `catalogKey`。
- 是否按里程提醒。
- 里程间隔。
- 是否按时间提醒。
- 时间间隔。
- 黄色阈值。
- 红色阈值。
- 创建时间。

### 9.3 车辆结构

每辆车包含：

- 车辆 ID。
- 品牌。
- 车型。
- 当前里程。
- 禁用项目 ID 列表。
- 上路日期。

每辆车下包含该车全部保养记录。

### 9.4 保养记录结构

每条保养记录包含：

- 记录 ID。
- 保养日期。
- 项目标识数组。
- 费用。
- 里程。
- 备注。

当前代码事实：

- 字段名为 `itemNames`，但导出内容实际优先为 `catalogKey`，没有 `catalogKey` 时才是项目名称。

重构时建议：

- 如果调整备份结构，应把项目字段改名为更准确的 `itemKeys` 或 `itemRefs`，并提供兼容旧备份的导入逻辑。

### 9.5 导入校验

恢复导入时必须校验：

- 车辆上路日期必须是严格 `yyyy-MM-dd`。
- 保养日期必须是严格 `yyyy-MM-dd`。
- 备份内车辆 ID 不能重复。
- 备份内保养记录 ID 不能重复。
- 车型配置不能重复。
- 车辆引用的车型配置必须存在。
- 当前规则下，同一品牌加车型不能出现多辆车。
- 车型内保养项目名称不能重复。
- 车型内非空 `catalogKey` 不能重复。
- 保养项目名称不能为空。
- 保养记录项目不能为空。
- 保养记录项目必须能映射到该车型配置中的项目。

### 9.6 项目映射

导入保养记录项目时：

- 优先按 `catalogKey` 匹配。
- 匹配不到时按名称匹配，用于兼容旧备份或自定义项目。
- 同一条记录内映射到重复项目 ID 时去重。
- 如果存在无法映射的项目，恢复失败。

## 10. 全局业务约束

### 10.1 日期

- 所有业务日期只表达年月日。
- 标准字符串格式为 `yyyy-MM-dd`。
- 如果实现层必须存储时分秒，固定为当天 `00:00:00`。
- 今天、车龄、提醒进度必须使用统一日期上下文。
- 手动日期开启时，业务今天等于手动日期。

### 10.2 唯一性

- 车辆：当前同一品牌加车型只允许一辆车。
- 保养记录：同一车辆同一日期只允许一条记录。
- 保养记录项目：同一车辆同一日期同一项目只允许出现一次。
- 保养项目配置：同一车辆内项目名称不能重复；默认项目使用 `catalogKey` 稳定识别。

### 10.3 数据隔离

- 保养项目按车辆隔离。
- 保养记录按车辆隔离。
- 保养提醒按当前应用车辆隔离。
- 保养记录页按当前应用车辆隔离。
- 删除车辆必须清理该车辆的数据边界内的记录和项目。

### 10.4 里程

- 车辆当前里程和保养记录里程均为整数公里。
- 新增或编辑保养记录后，如果记录里程大于车辆当前里程，则更新车辆当前里程。
- 记录里程小于或等于车辆当前里程时，不回退车辆当前里程。
- 里程提醒计算中，当前里程小于上次保养里程时，已使用里程按 0 处理。

### 10.5 费用

- 保养记录费用为非负数。
- 输入最多允许两位小数。
- 未填写或非法费用不能保存，除非当前场景费用只读并复用原记录费用。

### 10.6 删除

- 删除车辆会清理车辆、保养记录和保养项目配置。
- 删除整条保养记录会清理该记录的项目关系。
- 删除项目维度记录时，可能只从记录中移除项目，也可能删除整条记录。
- 有历史记录的自定义项目不能删除。
- 删除保养记录不会自动重算或回退车辆当前里程。

## 11. 重构验收清单

后续重构版本至少应通过以下业务验收：

- 可以新增车辆，并为车辆生成独立保养项目配置。
- 同一品牌加车型重复添加时会被阻止，除非新 PRD 明确改变该规则。
- 当前应用车辆切换后，保养提醒和保养记录同步切换范围。
- 新增同车同日保养记录会被阻止。
- 同一保养记录内重复选择同一项目不会形成重复关系。
- 新增保养记录后，车辆当前里程只增不减。
- 同时配置里程和时间提醒时，提醒进度采用更高进度。
- 进度达到 `100%` 显示黄色状态，达到 `125%` 显示红色状态。
- 自定义项目有历史记录时不能删除。
- 禁用项目不出现在新增保养记录和保养提醒中。
- 按项目维度删除多项目记录中的某一项时，只移除该项目，不误删整条记录。
- 按项目维度跨日期编辑多项目记录时，会拆分出新记录。
- 备份文件中的日期全部为 `yyyy-MM-dd`。
- 恢复备份时先清空再导入，失败时不留下半导入数据。
- 导入项目映射优先使用 `catalogKey`，不能依赖展示名称作为默认项目稳定标识。
